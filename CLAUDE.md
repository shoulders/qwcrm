# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What this is

QWcrm is a standalone PHP CRM for small businesses: clients, work orders (jobs), scheduling, invoicing, payments, credit notes, vouchers, expenses, suppliers, and financial reporting, all tied together in one app. It runs on a bespoke framework ("qframework") descended from Joomla 3.x internals, not a modern PHP framework — there is no router package, no ORM, and no dependency injection container beyond a handful of static factories.

PHP 7.4+ and MySQL 5.0+. No JS build step / bundler / package.json — this is server-rendered PHP + Smarty templates with plain CSS/JS assets under `src/themes/`.

## Working environment

- Application root for anything path-relative (config, includes, `.htaccess`) is `src/`, not the repo root.
- `src/configuration.php` (git-ignored) holds the live DB/site config; it's generated by the setup wizard (`src/setup/install/`). There is no `.env.example` — inspect `src/setup/install/install_configuration.php` and the `QConfig` class it writes to see available keys.
- There is no composer `scripts` section, no test suite, no linter, and no CI workflow configured in this repo. Don't invent `composer test` / `npm run lint` commands — none exist. Verify behavior by tracing the code path and, where possible, exercising it through a running instance.
- Dependencies are managed by Composer (`composer.json`/`composer.lock`) but install to `src/libraries/vendor/` (see `config.vendor-dir`), not the default `vendor/`.
- Debugging is via Xdebug (`.vscode/launch.json` has "Listen for Xdebug" and "Launch Built-in web server" configs, port 9003).

## Request lifecycle

Entry point is `src/index.php`. Flow per request:

1. `libraries/qframework/includes/loader.php` bootstraps: loads `configuration.php`, defines path/URL constants, requires vendored Joomla-derived session/auth/user/crypto classes, then Composer autoload, then `Factory.php` and `CMSApplication.php`.
2. `CMSApplication::classFilesLoad()` scans directories and auto-registers every instantiable class in them onto `$app` by lowercased class name — this is how `$app->components->workorder`, `$app->system->page`, etc. come to exist. It's run once each for `qframework/Sections/` (→ `$app->sections`... actually wraps `system`/`components`), `qframework/System/` (→ `$app->system->*`), `src/components/_includes/` (→ `$app->components->*`), plus `modules/` and `plugins/` (present in the framework but unused by this app — both directories are empty).
3. `Factory::getApplication('site')` builds the app; `$app->execute()` runs it.
4. `System\Router::pageController()` resolves the request to a `component`/`page_tpl` pair (from `?component=X&page_tpl=Y` or, when SEF/pretty URLs are enabled, from the path segments) and checks: maintenance mode → link validity → page-exists → **ACL** (looked up per-request against the `PRFX.user_acl_page` DB table, keyed by usergroup + `"component:page_tpl"`) → falls back to `core:404` / `core:403` / `core:dashboard` / `core:home` as appropriate.
5. The resolved controller file `src/components/<component>/<page_tpl>.php` is `include`d as a plain script (not invoked as a method) — inside it, `$this` is the calling `Components`-derived object, so it can call `$this->app->...` directly. These files just gather data and `$this->app->smarty->assign(...)` it; they don't return output.
6. Smarty renders the matching template under `src/themes/<theme>/templates/<component>/<page_tpl>.tpl` (theme name from config, default `default`).
7. Cron: `src/cron.php` is a CLI entry that just requires `index.php` with `_REAL_CRONJOB` defined; `$app->components->cronjob->systemRun()` also runs a "pseudo-cron" opportunistically on normal web requests when due.

## Code layout

- `src/components/<name>/` — page controllers, one file per action: `new.php`, `edit.php` (often folded into `new.php`), `details.php`, `search.php`, `overview.php`, `status.php`, `print.php`, `email.php`, autosuggest/ajax endpoints, etc. The file name **is** the `page_tpl` and must match the ACL entry and the `.tpl` template name.
- `src/components/_includes/<Name>.php` — one business-logic class per domain (`Workorder.php`, `Invoice.php`, `Client.php`, `Payment.php`, `Voucher.php`, `Expense.php`, `Creditnote.php`, `Report.php`, `User.php`, `Company.php`, `Core.php`, `Administrator.php`, `Setup.php`, `Help.php`, plus `CoreMenu.php`/`CoreTheme.php`). Each extends the abstract `Components` (in `libraries/qframework/Sections/Components.php`), which just gives it `$this->app`. These hold the `getRecord()`/`getRecords()`/`getStatuses()`/`getHistory()` style data-access methods that controllers call — this is the closest thing to a model layer; SQL is written inline with ADOdb (`$this->app->db->execute($sql)`, `$this->app->db->qStr(...)`), not an ORM/query builder.
- `src/components/payment/methods/PaymentMethod*.php` and `payment/types/PaymentType.php` — strategy-style classes for the different payment methods (cash, cheque, card, PayPal, direct debit, bank transfer, voucher, credit note, other).
- `libraries/qframework/System/*` — core framework services (`Router`, `Page`, `Security`, `General`, `Variables`, `Email`, `Pdf`, `Communication`, `QError`) reachable as `$app->system-><lowercasedClassName>`.
- `libraries/qframework/Sections/*` — the abstract base classes (`Components`, `System`) that everything else extends.
- `libraries/qframework/joomla/` — vendored/adapted Joomla 3.x subsystems (session, input filtering, crypto, authentication, user) kept because qframework grew out of Joomla and still leans on these for session/auth handling. Treat as third-party code; avoid modifying unless fixing a framework-level bug.
- `libraries/custom/smarty/` — custom Smarty plugins.
- `themes/<name>/templates/<component>/<page_tpl>.tpl` — Smarty view per controller/page.
- `setup/install/`, `setup/migrate/`, `setup/upgrade/<version>/` — first-run installer, MyITCRM data migrator, and versioned upgrade steps. Each upgrade version folder pairs `upgrade_database.sql` (schema/data changes) with `upgrade_routines.php` (PHP-side migration logic run in sequence by the setup wizard) — follow this pairing when adding a new upgrade step for a version bump.
- `language/en_GB/LC_MESSAGES/site.po`/`.mo` — gettext translation catalog; strings are wrapped in `_gettext("...")` in PHP. `language/_gettext_only/*.php` are string tables that exist purely so `xgettext`-style tooling picks up otherwise-dynamic strings (page names, system variable labels) — update these alongside any new translatable string that isn't a literal `_gettext()` call.

## Conventions to follow

- New pages: add the controller under `src/components/<component>/<page_tpl>.php`, the matching Smarty template under `themes/default/templates/<component>/<page_tpl>.tpl`, and an ACL row in `user_acl_page` for `"<component>:<page_tpl>"` (via the relevant `setup/upgrade/<version>/upgrade_database.sql` for existing installs) — a page with no ACL row is treated as inaccessible (403) even for admins.
- Business logic belongs in `components/_includes/<Name>.php`, not in the controller file — controllers should stay thin (fetch via the `_includes` class, `assign` to Smarty).
- DB access goes through ADOdb (`$this->app->db`); always use `$this->app->db->qStr()` to quote values interpolated into SQL (no parameterized-query layer exists here, so this is the injection guard in use throughout the codebase).
- User-facing strings should be wrapped in `_gettext(...)` for translation.
- Schema/data changes that need to ship to existing installations must go through a new `setup/upgrade/<version>/` step, not just an install-time SQL change, or upgrading users' databases will drift from fresh installs.
